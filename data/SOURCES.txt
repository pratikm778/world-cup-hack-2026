--------------------------------------------------------------------------------
[1] POLYMARKET GAMMA API — market discovery
--------------------------------------------------------------------------------

  Base URL    : https://gamma-api.polymarket.com
  Auth        : None (public)
  Provider    : scripts/providers/polymarket.py :: PolymarketProvider

  Endpoints used:
    GET /events?slug=<slug>
      Returns: [event] — a single event with .markets[] (each market has
               .question, .clobTokenIds, .outcomePrices, .volumeNum,
               .groupItemTitle, .sportsMarketType).

  Sibling-event slug pattern for EPL games (locked):
    {base}                       3 markets   moneyline (Home/Draw/Away)
    {base}-more-markets          9 markets   spreads + totals + BTTS
    {base}-halftime-result       3 markets   halftime moneyline
    {base}-exact-score          24 markets   score grid (7 are unfunded shells)
    {base}-total-corners         5 markets   corners over/under

  Slugs we probed and confirmed do NOT exist for EPL regular-season games:
    -total-cards, -total-goals, -total-shots, -anytime-scorer,
    -first-scorer, -double-chance, -btts

  Quirks:
    - Unfunded shell markets (volume = $0) have no `clobTokenIds` field. We
      skip them silently — there's no orderbook to query.
    - Player anytime-scorer markets do NOT exist as match-tied events for
      regular EPL games. Polymarket UI shows generic player chips that are
      not Gamma-queryable for a specific match.
    - The /public-search endpoint exists but ranks poorly; trust slug probes.
    - `negRiskMarketID` does NOT cluster sibling events — sibling events share
      a slug prefix, not a neg-risk ID.

[2] POLYMARKET CLOB API — price history & live prices
--------------------------------------------------------------------------------

  Base URL    : https://clob.polymarket.com
  Auth        : None for read paths (orders require auth — we don't write)
  Provider    : scripts/providers/polymarket.py :: PolymarketProvider

  Endpoints used:
    GET /prices-history?market=<token_id>&startTs=<unix>&endTs=<unix>&fidelity=<minutes>
      Returns: { history: [ { t: <unix>, p: <0..1 price> }, ... ] }

  CRITICAL — `fidelity` units:
    `fidelity` is in MINUTES, not seconds. Third-party docs (medium,
    dlthub, pm.wiki) disagree; the empirical truth is verified:
        fidelity=5  over 240min window → 50 buckets  (1 per 5 min)
        fidelity=60 over 240min window →  4 buckets  (1 per 60 min)
        fidelity=1  → densest available (~1 point per minute, server-capped)

  Response cap:
    Server caps responses around 250–330 points. For wide windows + low
    fidelity, you trade granularity for coverage. We split each token into
    two pulls (pre-match + in-match) to avoid the cap biting either window.

  `interval=max|1d|1w|1m` works ONLY for active markets. Closed games (our
  whole demo set) require explicit startTs / endTs.

  Token IDs are ERC-1155 token IDs. Each Yes/No outcome is its own token.
  For one market with outcomes ["Yes","No"], you get 2 token IDs in
  `.clobTokenIds`.

  Future live mode (NOT YET WIRED):
    - REST poll on midpoint: GET /midpoint?token_id=<id>
    - WebSocket for sub-second updates: wss://ws-subscriptions-clob.polymarket.com

[3] ESPN SOCCER SUMMARY API — commentary + structured events
--------------------------------------------------------------------------------

  Base URL    : https://site.api.espn.com
  Auth        : None (public, undocumented but stable)
  Provider    : scripts/providers/espn.py :: EspnProvider

  Endpoints used:
    GET /apis/site/v2/sports/soccer/{league}/summary?event=<gameId>
      League codes we support:
        epl       → eng.1
        laliga    → esp.1
        seriea    → ita.1
        bundesliga → ger.1
      Response top-level keys (we use): commentary, keyEvents, header

  Same endpoint serves historical AND live in-progress games. The
  `commentary` array grows tick-by-tick as the match runs. This is why
  ESPN is our primary commentary source — same code works in both modes.

  Per-entry shape (the ones we care about):
    sequence  : int, monotonically increasing — our dedup key
    text      : narrative string ("Goal! Manchester City 2, Arsenal 1. ...")
    time.displayValue : "16'" or "90'+6'" — parsed into minute + extra_time
    play.type.type    : structured event type (goal, yellow-card, foul, …)
    play.wallclock    : UTC ISO timestamp — JOIN KEY against Polymarket
    play.athletesInvolved[].displayName : player names
    play.team.displayName               : team

  Cache header: `cache-control: max-age=1`. No ETag. Every poll is a full
  fetch. Payload is ~50–100 KB per match.

  Empirical publication cadence on Man City vs Arsenal 2026-04-19
  (81 of 96 entries have wallclock):
    median gap between entries : 51s
    p25                        :  2s   (bursts: card+foul same instant)
    p75                        : 108s
    p95                        : 326s  (5.4 min — typical halftime)
    max                        : 926s  (15 min — full halftime)

  Default polling cadence in stream_live(): 60 seconds.
    Tradeoff: events lag ~30s on average (half the interval); bursts get
    batched into one poll; gentle on ESPN's tolerance.

  Termination criterion for stream_live():
    Match completed (header.competitions[0].status.type.completed = true)
    AND one subsequent poll returned no new entries. Two-phase guard
    handles the race where "Match ends" appears in the same payload as
    the completion flag.

  15 of 96 entries have NO wallclock (framing texts like "Match ends, …"
  and "First Half ends, …"). Filter on wallclock != null when computing
  event density.

--------------------------------------------------------------------------------

  Producer  : pull_polymarket() — CLOB /prices-history
  Source    : 148 API calls (74 tokens × 2 windows: pre-match + in-match)
  Used for  : the price-time series the agent reacts to

  Per-file schema:

    question          : market question this token belongs to

    event_slug        : sibling slug
    windows.pre_match : { start, end, fidelity_minutes: 10, point_count }
    windows.in_match  : { start, end, fidelity_minutes: 1,  point_count }
    points            : [ { ts_utc, price, token_id }, ... ] sorted by ts_utc

  Two-window strategy:
    pre-match (13 days before kickoff)  : fidelity=10min → ~300 points
    in-match (30min pre to 90min post)  : fidelity=1min  → ~180 points
    Merged per token, sorted by wallclock.

  Total: ~115,700 price points across 74 tokens.

  Replay-engine hint:
    Don't open one file per token at request time. Pre-build a single
    timeline indexed by wallclock minute → all-token-prices-at-that-minute.
    Memory cost is trivial (~5MB resident); request cost drops to O(1).


================================================================================
PART 3 — JOIN KEYS (how the data sources connect)
================================================================================

  Wallclock UTC          ← THE primary join key
                           (commentary.ts_utc ↔ prices.points[].ts_utc)
                           Don't compute "minute since kickoff" yourself;
                           use the wallclocks ESPN gives you on each play.

  Match identity:
    fixture.polymarket_event_slug  → Polymarket Gamma
    fixture.espn_game_id           → ESPN summary
    No shared ID across providers. We bind them in build_fixture() and
    pass FixtureRef through. To add a new match, update build_fixture
    or accept the IDs via CLI args.

  Token → market → event:
    Each prices/<token_id>.json carries market_id + event_slug, so a
    price tick is fully self-describing without external joins.
